## 5. Module Internal Structure

Every Standard API module follows this internal structure:

```
modules/{module-name}/
├── index.ts          ← Function exports (primary interface)
├── server.ts         ← REST API server (generated by Milo pattern)
├── proxy.ts          ← Smart call wrapper (generated by Milo pattern)
├── {domain-logic}/   ← Business logic, split by concern
│   ├── {entity}.ts
│   └── {service}.ts
├── types.ts          ← Types and interfaces for this module
└── tests/
    ├── unit/
    └── integration/
```

Layer conventions:
- `index.ts` only exports — no business logic inside it
- `server.ts` only routes to functions — no logic inside handlers
- `proxy.ts` only routes based on DEPLOYMENT_MODE — no logic inside it
- All business logic lives in the domain-logic folder

## 9. Deployment Abstraction Standard

**This is mandatory for all Standard API modules.** Every module must implement the three-layer pattern:

1. **Function exports** (`index.ts`) — the business logic, exportable as a library
2. **REST API server** (`server.ts`) — same functions exposed as HTTP endpoints
3. **Smart call wrapper** (`proxy.ts`) — routes based on `DEPLOYMENT_MODE`

**Deployment flag:** `DEPLOYMENT_MODE` environment variable
- `monolith` — direct function calls (same process)
- `distributed` — REST API calls (separate service)

See `milo/knowledge/deployment-abstraction.md` for implementation patterns.
